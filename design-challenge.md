# AI 原生多模态活档案系统：从架构到落地的 12 次练习实训大纲

## 前言：三种客户的需求，一种架构

1）一个学院有 1000 学生，300 教职员工，要不断完善其事件和人物的活档案（照片，视频，论文，文章，学术和非学术活动的各种记录）， 并被学院的各种用户消费（增删改查，并生成各种报告或多模态的艺术展现）。另外，还有可能被学院内的智能体（如 openClaw 龙虾）调用并输出信息。这个学院会进一步收集和学校相关的学术界和工业界人物，合作的导师，等“相关”人物和发生关系的类型（合作，推荐学生，演讲，客座教授等）。这些信息只在本学院可见，人物的隐私信息只有部分管理人员可见。

2）一个历史悠久的校友会，有很多校友并写了很多回忆文章，并不断有校友聚会。他们希望把校友的文章、活动、聚会等信息形成丰富的档案并支持增删改查，并生成艺术的展现（文章，周年纪念，短视频等）。校友可以访问这个档案并不断提供新的回忆或者确认历史和人物的描述。智能体（如龙虾）也可以逐步爬取网络数据，收集校友信息，核对之后，加入校友会文档。

3）一个大家庭，有老少 4 辈人，也有很多照片视频和回忆，这些信息少量是文章大量是语言描述，例如一个老人和孙子一问一答地谈论老照片中的细节。家庭成员大部分都用手机（不同品牌），希望通过手机贡献更多照片，更多语音评论，但是看到成体系的语音+文字对话。同样，家庭成员也希望看到艺术方式展现人物的足迹，某张照片的周年纪念，并分享到外部或者家庭的显示设备上。

我们不会手动设计 3 种截然不同的架构来实现这三类需求，而是用一种架构来解决，同时希望达到功能模块的高度重用。

## 一、 三种客户的需求与产品本质

本项目旨在为三种截然不同的场景构建多模态活档案系统。表面上服务于不同的 B 端（学院、校友会）和 C 端（大家庭）客户，但核心痛点高度一致：**非结构化多模态记忆的沉淀、关联与再创作。**

### 1. 场景剖析

* **场景一：高频生长的“创新引擎”—— 现代学院活档案**
    * **用户画像**：1000 名学生（高频活跃的流转节点）、300 名教职员工（知识产出者与管理者）。
    * **输入特征**：高频、多模态、强专业性。包含内生数据（论文、PPT、合影）与外延数据（合作导师、客座教授信息）。
    * **核心诉求**：构建带有**严格权限控制（RBAC）**的“科研脉络”与“人才图谱”。隐私与敏感关联仅限管理员可见。
    * **输出消费**：权限隔离的增删改查；供内部智能体（OpenClaw 龙虾）调用的标准接口；一键生成学术与日常混剪的汇报视频。
* **场景二：跨越时空的“情感纽带”—— 历史校友会活档案**
    * **用户画像**：跨越多个届别、遍布全球的校友，存在大量“失联”状态。
    * **输入特征**：低频集中、文本重度、岁月痕迹重。回忆录、老照片，以及**外部网络爬取**的公开校友动态。
    * **核心诉求**：实体消歧（认人）与记忆唤醒。将模糊的老照片、回忆录文本与网络爬虫（智能体）抓取的最新履历精准关联。
    * **输出消费**：校友通过搜索拼凑班级碎片记忆，并在校庆时生成以个人为主角的专属青春交互式纪录片。
* **场景三：随时随地的“口述历史”—— 四世同堂的家族档案**
    * **用户画像**：具有较高技术壁垒的老年人（倾诉者）和习惯移动互联网的年轻一代（记录者）。
    * **输入特征**：移动端优先、语音为主、对话驱动。以“拿着老照片录音问答”为典型交互方式。
    * **核心诉求**：极低阻力的采集与口述历史结构化。系统需直接从方言或口语对话中提取人物关系和历史背景。
    * **输出消费**：自动整理成“有图有声有文字”的家族时间线；支持将生成的家族传记短片或动态画册**一键投屏分享至外部显示设备（智能电视/相框）**。

### 2. 产品抽象：SaaS/PaaS 平台化切入点

本实训提取产品最大公约数，构建统一的**“AI 原生多模态图谱引擎 (Multimodal Graph RAG)”**：

* **统一的底层数据结构 (The Universal Ontology)**：底层统一抽象为 `[Person]`、`[Event]` 节点，论文、录音作为挂载的 `[Artifacts]`。
* **统一的“无感”采集引擎 (The Ingestion Layer)**：底层调用同一套处理管道：识别人脸 + 语音转文本 + 提取实体 + 写入图数据库。
* **统一的交互范式 (Human-in-the-loop)**：通过自然对话进行后台数据纠错与补充。
* **统一的创意引擎 (The Generative Layer)**：Agent 检索精准素材构成故事线，生成 Prompt 调用音视频大模型。

## 二、 核心架构定调的 5 个关键决策

为确保工程质量，系统设计需严格遵循以下 5 个物理边界与架构规范：

1.  **底层存储的“三足鼎立”与权限隔离**
    * **对象存储 (S3/OSS)**：存放原始文件（照片、视频、长文 PDF）。
    * **关系型数据库 (PostgreSQL)**：存放账号体系与**基于角色的权限控制 (RBAC) 策略**。这是保障学院场景中“隐私信息仅管理人员可见”的核心屏障。
    * **向量数据库 (Vector DB)**：存放高维特征向量，作为实现“同名消歧”和“模糊搜索”的核心引擎。
2.  **关系数据的图谱化 (Graph RAG)**
    * 必须使用图数据库（如 Neo4j）存储实体节点和复杂的关系边（如“客座教授”、“推荐学生”）。严禁使用 Markdown 等无结构文本存储网络关系。
3.  **Voice-First Hybrid UI 前端交互**
    * 前端形态必须是“对话与传统 UI 的深度融合”。长者端提供“大按钮语音+照片瀑布流”；学院/校友端提供“时间轴组件+悬浮智能对话框”及大屏投屏协议。
4.  **确定性与非确定性模块的通信隔离**
    * **传统 Web Service (RESTful/gRPC)**：处理系统内部确定的数据流转（鉴权、文件上传）。
    * **MCP (Model Context Protocol) / Skill**：专用于 AI 智能体任务流，为大模型提供受控的数据库查询权限。
5.  **OpenClaw（及“龙虾”等衍生智能体）的精准定位**
    * **禁止项**：不可作为系统高并发后端。
    * **适用项**：作为系统边界外的**主动式外挂**。例如部署在本地的 OpenClaw 用于调用 Premiere 剪辑本地重型视频；或者作为“龙虾爬虫 Agent”在互联网上静默游走，抓取校友公开信息后，通过标准 API 喂给中央图谱。

## 三、 12 次 MVP 冲刺实训大纲 (The 12-Session Sprints)

由资深 DB 工程师和前端工程师作为 Mentor 带领，3 组具有初级 Vibe Coding 经验的学生团队将在 12 次实训中各自开发出 1 个 MVP。**核心原则：Mentor 钉死接口与表结构，学生利用 Vibe Coding 冲刺业务逻辑与前端展现。**

### 阶段一：底层基建与数据建模 (Infrastructure & Ontology)

* **Session 1：产品宣发与架构破冰**
    * **技术要点**：拆解 3 个场景的共性，发布《架构白皮书》。明确“一个底层引擎，三套交互皮肤，多个独立 Agent”的协同模式。
    * **学生任务**：配置开发环境，跑通 Cursor/Windsurf 基础工作流。
* **Session 2：数据库搭建与 API 契约（引入 RBAC）**
    * **技术要点**：演示 PostgreSQL + 对象存储 + 向量数据库的集成。重点讲解如何设计基于角色的权限控制（User-Role-Permission）来满足学院场景的隐私需求。
    * **学生任务**：利用 Vibe Coding 生成后端基础 CRUD RESTful API，并实现带 JWT 鉴权的本地至对象存储的上传链路。
* **Session 3：万物互联的图数据库设计 (Neo4j)**
    * **技术要点**：设计通用的本体结构。定义 `Person`, `Event`, `Media` 节点。扩展复杂关系边：`[Person]-[:GUEST_PROF_AT]->[Institution]`, `[Person]-[:RECOMMENDED]->[Person]`。
    * **学生任务**：在 Neo4j 建立测试图谱。编写核心 Cypher 查询，例如：“查找张教授近两年推荐过的所有学生及其参与的科研事件”。

### 阶段二：AI 注入与知识抽取 (Data Ingestion & Agentic Extraction)

* **Session 4：多模态感知管道 (Vision & Audio)**
    * **技术要点**：接入 Whisper API (语音转文本) 与视觉大模型 API (提取照片描述与特征)。
    * **学生任务**：编写 Python 脚本。输入一张老照片和一段方言语音，输出包含时间、地点、潜在人物信息的结构化 JSON。
* **Session 5：信息整理助理 Agent (The Archivist)**
    * **技术要点**：引入 LangGraph 构建无头智能体。讲解实体抽取 (NER) 与同名消歧策略。
    * **学生任务**：实现非结构化数据管道——Agent 读取会议记录/口述录音，自动提取三元组，并结合向量相似度判断是否为图谱中已存在的人物，最终写入 Neo4j。
* **Session 6：MCP 与安全工具调用**
    * **技术要点**：如何将 PostgreSQL 的权限校验和 Neo4j 的查询封装为标准的 MCP Tools。防止 Agent 越权查询隐私数据。
    * **学生任务**：为 Archivist Agent 装备“查重”技能。Agent 在插入新实体前，必须先调用 MCP 工具查询图谱，确认无重复后再执行写入。

### 阶段三：前端交互与 Vibe Coding 冲刺 (User Experience)

* **Session 7：统一组件库与 3 套 UI 皮肤**
    * **技术要点**：Mentor 下发底层 API 列表。讲解响应式设计与设备投屏协议（DLNA/AirPlay）。
    * **学生任务 (Vibe Coding)**：
        * 组 1（学院）：生成带细粒度权限开关和复杂数据透视表的 Web Dashboard。
        * 组 2（校友）：生成极具岁月感的时间轴 (Timeline) H5 页面。
        * 组 3（家庭）：生成包含巨大“按住说话”按钮的移动端界面，并集成局域网内一键投屏至智能电视的按钮。
* **Session 8：对话驱动的交互 (Human-in-the-loop)**
    * **技术要点**：打通前端语音 -> 转文本 -> Agent 意图识别 -> 图谱更新的全链路。
    * **学生任务**：实现“上下文智能追问”。当老人上传照片但未说明时间时，UI 悬浮框自动弹出 Agent 提问：“这张大合影看起来是夏天拍的，您还记得是哪一年吗？”
* **Session 9：Graph RAG 检索与权限级可视化问答**
    * **技术要点**：Text-to-Cypher 技术。结合 Session 2 的权限系统，确保 Graph RAG 只检索当前用户有权查看的子图。
    * **学生任务**：实现自然语言搜索。如输入“找找当年实验室的聚餐照片”，系统需翻译为图查询语句，渲染出照片瀑布流。

### 阶段四：艺术生成、外部智能体联通与部署 (Generative Art & Ops)

* **Session 10：创意艺术家 Agent (The Creator)**
    * **技术要点**：利用图谱检索出的“故事线时间轴”组装超长 Prompt，调用高保真视频/音频生成大模型（如 Veo、Suno）。
    * **学生任务**：实现“艺术展现”功能。用户勾选某几段校友聚会记录，系统自动生成包含转场、配乐和 AI 旁白提示词的回忆短片工程配置。
* **Session 11：OpenClaw/龙虾 Agent 外部接入实战**
    * **技术要点**：讲解中心化后端与外部自治智能体的协作边界。
    * **学生任务**：
        * 开发“龙虾 (Lobster) 爬虫 Agent”：模拟在外部网络抓取某知名校友的公开履历，通过数据清洗后，调用系统的 OpenAPI 申请合并至中央图谱。
        * 编写 OpenClaw 桌面指令：授权 OpenClaw 通过 API 查询中央图谱数据，并在本地终端生成统计报告。
* **Session 12：独立部署、测试与 Demo Day**
    * **技术要点**：Docker 容器化、跨域资源共享 (CORS) 与云服务器 (AWS/阿里云) 部署实操。
    * **学生任务**：各组完成系统（DB + 后端 Agent 容器 + 前端静态托管）打包上线，执行交叉测试（重点测试隐私越权漏洞），并进行最终的成果路演。

## 四、 教学成果与技术价值观总结

通过这 12 周的高强度工程演练，学生不仅各自交付了具备商业雏形的 MVP，更建立了一套成熟的**AI 软件工程（AI-Native Software Engineering）方法论**。他们深刻理解了各项新兴技术的优缺点，避免了为了“炫技”而滥用技术的陷阱：

1.  **重新定义 Vibe Coding 的边界**：学生充分体验了利用 AI 辅助编程（如 Cursor）一键生成前端 UI 和样板代码的极速快感。但由于 Mentor 在前期（Session 2 & 3）严格锁死了 API 契约和数据库表结构，学生深刻意识到：**AI 可以代写代码，但不能代做架构设计**。扎实的软件工程规范是防止 Vibe Coding 演变成不可维护的“面条代码”的唯一屏障。
2.  **认清“确定性”与“非确定性”的界限**：学生学会了不滥用对话框。他们意识到，对于收集表单等确定的结构化输入（如学院录入教职工工号），传统 Web UI 依然是效率之王；纯文本聊天机器人并不适合展示复杂的时间轴和人物脉络。因此，系统最终走向了“传统 UI 负责展示与精准控制，AI 对话框负责模糊意图处理与信息补全”的混合体验架构。
3.  **MCP 与 Skill 的精准应用**：通过引入 MCP（Session 6），学生理解了如何让大模型与真实的工程世界安全对话。不再是盲目地给大模型开放全局后门，而是通过标准协议暴露受限的查询“工具（Skills）”，将 AI 真正规训为业务流程中的一环。
4.  **扎实的 Cloud-Native 数据底座**：跨越了玩具级单机应用的局限。学生熟练掌握了将对象存储（S3）、关系型数据（PostgreSQL 的权限 RBAC 控制）与非关系型数据（Vector/Neo4j 的图谱推理）解耦融合的“三足鼎立”架构，具备了企业级 Cloud 基础。
5.  **克制而精准地使用 OpenClaw 等独立 Agent**：打破了“万物皆可集成在单一代码库”的迷思。学生明确了 OpenClaw 与“龙虾”这类自主智能体的生态定位——它们不在系统的中央服务器内，而是作为**边缘侧的执行者或数据觅食者**。它们唯一的、无可替代的价值在于打通**物理环境/外部广域网与云端中央图谱**的最后一公里。

此课程设计旨在确保技术创新始终服务于真实业务逻辑，在深刻理解技术本质的基础上，进行克制、精准且富有创造力的架构选型。
