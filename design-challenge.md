## 三种客户的需求
 1）一个学院要不断完善其事件和人物的活档案（照片，视频，论文，文章，学术和非学术活动的各种记录）， 并被学院的用户消费（增删改查，并生成艺术展现）。  
 2）一个历史悠久的校友会， 有很多校友并写了很多回忆文章，并不断地有校友聚会， 也要形成活档案并做增删改查，并生成艺术展现。 校友可以访问并不断提供新的回忆或者确认历史和人物的描述。  
 3）一个大家庭， 有老少 4 辈人， 也有很多照片视频和回忆，这些少量是文章大量是语言描述， 例如一个老人和孙子一问一答地谈论老照片中的细节。 家庭成员大部分都用手机（不同品牌）， 希望通过手机贡献更多照片，更多语音评论，但是看到成体系的语音+文字对话。   
 
 我们不会手动设计 3 中不同的架构来实现这三类需求， 而是用一类架构来解决。 

正是绝佳的**“SaaS / PaaS 平台化”**切入点。

表面上看，我们在服务三种截然不同的 B 端（学院、校友会）和 C 端（大家庭）客户；但透过现象看本质，我们在解决的是同一个核心痛点：非结构化多模态记忆的沉淀、关联与再创作。

站在产品设计的角度，让我们先剥离具体的业务外衣，用统一的产品语言来透视这三个场景的“用户故事（User Story）”和“核心价值”。

场景一：高频生长的“创新引擎”—— 现代学院活档案
这是一个处于高速运转和扩张期的科研与教学综合体。

用户画像： 100名研究员（知识产出者）、500名学生（高频活跃的流转节点）、100名教职员工（管理者）。

输入特征（Input）： 高频、多模态、强专业性。每天都有大量的科研论文（PDF）、组会汇报（PPT）、学术演讲（音视频）以及活动合影（图像）产生。

核心诉求： 构建动态的“科研脉络”与“人才图谱”。学院需要实时追踪“谁在什么时候参与了什么课题”、“哪些技术理念在历代学生中传承”。

输出消费（Output）：

增删改查： “帮我调出张教授过去两年指导过的所有参与过大模型项目的学生名单及他们的答辩视频。”

艺术展现： 在年度总结或迎新大会上，一键生成一段混剪了核心科研突破瞬间和师生日常笑脸的震撼开场视频。

场景二：跨越时空的“情感纽带”—— 历史校友会活档案
这是一个具有深厚历史沉淀，但物理空间高度分散的熟人网络。

用户画像： 跨越几十个届别、遍布全球的校友。年龄跨度大，且很多人处于“失联”或“半失联”状态。

输入特征（Input）： 低频集中、文本重度、岁月痕迹重。海量的长篇回忆录文章、泛黄的老照片扫描件，以及偶尔举行的大型校友聚会所产生的新鲜影像。

核心诉求： 实体消歧（认人）与记忆唤醒。最大的痛点是如何把回忆录里提到的“睡在我上铺的兄弟小王”，和一张80年代模糊毕业照里的人脸，以及今天已经白发苍苍的“王总”精准地关联起来。

输出消费（Output）：

增删改查： 校友可以通过简单的搜索，拼凑出自己当年班级的所有碎片记忆。

艺术展现： 在百年校庆时，生成一部“光阴的故事”交互式纪录片，让每一位扫码观看的校友都能看到以自己为主角的专属青春回忆分线。

场景三：随时随地的“口述历史”—— 四世同堂的家族档案
这是一个极度重视隐私、情感羁绊最深，但技术门槛要求最低的场景。

用户画像： 具有极高技术壁垒的老年人（倾诉者）和习惯移动互联网的年轻一代（记录者/提问者）。

输入特征（Input）： 移动端优先、语音为主、对话驱动。这是最生动的输入方式：孙子拿着手机里的一张老照片，开启录音，问：“爷爷，这张照片里站在您旁边的这个阿姨是谁？”爷爷用方言或口语化的语音回答。

核心诉求： 极低阻力的采集与口述历史的结构化。不需要任何人去填表，系统必须能直接从对话录音中“听懂”人物关系和历史背景。

输出消费（Output）：

增删改查： 将杂乱的日常对话自动整理成“有图有声有文字”的精美家族时间线（Timeline），并在全家人的手机端同步更新。

艺术展现： 在老人寿辰时，将这些日常的语音对话和老照片融合，生成一段温暖的家族传记短片或带有配乐的有声数字画册。

产品抽象：为什么我们可以用一套架构“包打天下”？
作为产品经理，要说服研发团队（尤其是像您这样深谙软件工程与架构设计的领导者）不去做定制化开发，我们需要抽象出这三个场景的产品最大公约数。

这三个场景，本质上都在跑同一个**“AI 原生多模态图谱（AI-Native Multi-modal Graph RAG）”**的产品飞轮：

统一的底层数据结构（The Universal Ontology）：
不管是科研泰斗、老校友还是爷爷，在底层都是一个 [Person] 节点。不管是顶级学术会议、毕业典礼还是年夜饭，底层都是一个 [Event] 节点。不管是论文摘要还是口述录音，都是挂载在这些节点上的 [Artifacts]。

统一的“无感”采集引擎（The Ingestion Layer）：
不论是学院 Web 端上传的批量文档，还是家庭群里发的一段 60 秒语音，底层都在调用同一套多模态大模型处理管道：识别人脸 + 语音转文字 + 提取实体（人/事/时/地） + 写入图数据库。

统一的交互范式：对话式微调（Human-in-the-loop）：
不论面对谁，系统的“纠错”方式都是自然的对话。系统在后台发现冲突或缺失，就会通过前端（如微信、专属 App 或 Web 助手）主动提问：“这张脸我匹配不上，请问这是谁？” 用户的一句回复，就是在做最高效的数据标注。

统一的创意引擎（The Generative Layer）：
“艺术展现”不再是预设的视频模板，而是 Agent 拿着检索到的精准素材（节点连线构成的时间轴和故事线），写出 Prompt，去调用底层的视频生成大模型（如 Veo）和音乐生成大模型。

产品结论：

我们不需要为学院、校友会和家庭开发三套软件。我们需要开发的是一个**“多模态记忆编织引擎”。交付给客户时，底层架构完全一致，我们只是通过配置不同的前端交互界面（Web / 微信 / 独立 App），以及配置不同性格和权限的“数字助理 Agent”与“创意 Agent”**，就完成了产品的商业化矩阵。

## 如何实现
架构定调：

1. 原始数据的存储：关系型数据库 (PostgreSQL) 还是其他？
您的理解只对了一半。 现代多模态系统的底层存储必须是**“三足鼎立”**的混合架构，不能单靠 PostgreSQL 打天下：

对象存储 (Object Storage, 如 AWS S3 / 阿里云 OSS)：专门存放真正的原始文件（照片原图、视频 MP4、录音 WAV、长文 PDF）。数据库里只存这些文件的 URL 链接。

关系型数据库 (PostgreSQL)：存放结构化的高价值元数据。例如：账号体系、权限控制、机构的基础组织架构、日志记录。

向量数据库 (Vector DB，或使用 PostgreSQL 的 pgvector 插件)：这是实现“同名消歧”和“模糊搜索”的灵魂。人脸特征提取后会变成高维向量存放在这里，一篇文章的语义段落也会变成向量。当找“长得像李明的人”时，本质上是在这里做向量相似度计算。

2. 中间关系数据：图数据库 (Graph DB) 还是 Markdown？
绝对必须使用图数据库（如 Neo4j）。 用 Markdown 描述关系是一场工程灾难。

为什么不用 Markdown？ Markdown 是无结构的文本。如果用 Markdown 记录“张三在 1998 年和李四参加了校庆”，当系统需要查询“张三的所有同学参加过的所有活动”时，系统只能把所有 Markdown 扔给大模型去硬读，这会瞬间撑爆 Token 限制并导致严重的幻觉。

正确的做法 (Graph RAG)：实体（人、地点、事件）和它们之间的关系（参加、指导、亲属）作为节点和边严格存储在图数据库中。只有当用户提问时，系统会通过查询语句（Cypher）从图数据库中精确切出一小块相关的“子图”，在最后一刻将这个“子图”翻译成一段包含上下文的 Markdown 或 JSON，喂给大模型（LLM）去生成最终的流畅回答。

3. 前端界面：Vibe Coding 还是对话框？老人怎么用？
这里需要理清两个概念：“Vibe Coding” 是开发方式，“对话框/UI” 是产品形态。

您可以利用 Vibe Coding（如使用 Cursor 或 Windsurf）极快地开发出这个跨端系统。但针对这三个场景（尤其是老人），产品形态必须是“对话与传统 UI 的深度融合 (Voice-First Hybrid UI)”：

对于老人（输入层）：绝对不要纯文本对话框。界面核心应该是一个巨大的“按住说话/拍照录音”按钮。老人点开一张老照片，按住按钮说：“这是你大姑，当年在老宅院子里拍的。” 语音传到后台，大模型（如 Whisper）自动转录并提取人物关系，静默写入图谱。

对于所有用户（消费层）：纯聊天机器人是无法优雅地展示“艺术展现”和“历史脉络”的。 系统必须有传统的 UI 组件，如瀑布流照片墙、可缩放的家族/校友时间轴（Timeline）、以及可视化的关系网络图。对话框只是作为一个悬浮的“智能副驾（Copilot）”存在，用于处理复杂的筛选（“帮我过滤出只有二爷爷的照片”）。

4. 模块间的关系：MCP/Skill 还是传统的 Web Service？
两者都要，各司其职，切忌混用。

传统 Web Service (RESTful API / gRPC)：用于系统内部的确定性通信。比如前端传照片给后端、后端调用对象存储、用户登录鉴权。这些是软件工程的地基，不需要也不应该让 AI 参与决策。

MCP (Model Context Protocol) / Skill：这是专门给 AI 智能体（Agent）配备的“手和眼睛”。只有在非确定性的任务流中才使用。比如，后端的“整理档案 Agent”被唤醒后，它不知道接下来该干嘛。系统通过 MCP 向它暴露了一个 query_graph_database 的工具。Agent 思考后决定调用这个 MCP 工具去图数据库里查数据。

5. OpenClaw：到底用来做什么，不做什么？
基于我们前面确定的云端多租户架构，我们可以给 OpenClaw 下达最终的“判决书”：

OpenClaw 绝对不做什么？

不作为系统的后端服务器。 它承担不了海量的高并发数据入库。

不面向终端用户。 学生、校友、爷爷奶奶的手机里没有任何 OpenClaw 的存在。他们只与 Web 页面、微信小程序或 App 交互。

OpenClaw 用来做什么？（管理员的本地外挂）

本地素材的中转站：您（系统管理员或学院的宣传干事）的办公电脑里可能有几十个 G 的未经整理的本地院庆录像素材，还没传到云端。您可以让您电脑上的 OpenClaw 在夜里自动扫描这些本地硬盘，压缩转码后，批量推送到云端系统的“数据摄入 API”中。

调用本地重型生产力工具：如果云端的 Veo 视频生成 API 无法满足复杂的艺术展现需求，您需要用到本地电脑上的 Adobe Premiere 或 DaVinci Resolve。您可以对 OpenClaw 下令：“去云端图谱把李教授的故事线拉下来，然后在我的本地剪辑软件里按照时间轴生成一个基础工程文件。”

##如何落地
如何在 12 次练习中完成 3 个解决方案的 MVP，同时共享设计和模块？ 有资深 DB 工程师和前端工程师为 mentor， 10 个有 vibe coding 基础的学生作为 trainee。 
12 次 MVP 冲刺实训大纲。

第一部分：产品共识与本质（第 1 次课内容）
1. 三种客户需求的异同与产品本质

异同点：

学院（高频/重度）：关注科研脉络传承，数据并发量大，需严谨的文档解析能力。

校友会（低频/长尾）：关注历史记忆唤醒，需处理大量跨越年代的非结构化回忆文章和老照片。

大家庭（移动端/语音交互）：极简 UX，核心挑战是从口语化的多轮对话中自动抽取人物关系。

产品本质（最大公约数）：这三个系统底层共用同一套**“AI 原生多模态图谱引擎 (Multimodal Graph RAG)”**。我们交付的不是三个独立的软件，而是一个核心底座 + 三个不同性格的“数字助理 Agent” + 三套不同的前端交互皮肤。

第二部分：架构定调的 5 个核心决策（第 2 次课内容）
两位 Mentor 需要向学生明确系统的物理边界，禁止学生在 Vibe Coding 时随意发明架构：

原始数据层：坚决使用对象存储（S3/OSS）存放原文件，使用 PostgreSQL 存放账号、权限等结构化高价值元数据。

关系与推理层：必须使用图数据库（Neo4j）配合向量检索（Vector DB）来存储人物、事件的节点和边，绝不能用 Markdown 存关系。

前端交互层：采用 Voice-First Hybrid UI。老人用“大按钮语音+照片瀑布流”，学生和管理员用“Web 表单+悬浮对话框”。UI 使用 Vibe Coding 快速生成。

模块通信层：确定性的数据流转用传统 RESTful API/Web Service；非确定性的 AI 决策流转使用 MCP（Model Context Protocol）暴露工具给 Agent。

OpenClaw 的定位：不在终端用户手机里，不在后端服务器里。它只作为系统管理员/创作者个人工作站上的“桌面外挂”，用于将本地庞杂的素材批量推向云端流水线。

第三部分：12 次实训课件安排 (The 12-Session Sprints)
整个 MVP 周期分为四个阶段：基建、注入、交互、生成。两位 Mentor 负责 Review 接口，3 组学生通过 Vibe Coding 冲刺实现。

阶段一：底层基建与数据建模 (Infrastructure & Ontology)
Session 1：产品宣发与架构破冰

技术要点：拆解 3 个场景的共性。Mentor 发布架构白皮书（上述的第一、第二部分）。

学生任务：配置开发环境，熟悉 Cursor/Windsurf 工作流。

Session 2：数据库三足鼎立与 API 契约

技术要点：Mentor 演示如何搭建 PostgreSQL + 对象存储 + 向量数据库。

学生任务：利用 Vibe Coding 生成后端基础的 CRUD RESTful API，并打通本地到对象存储的文件上传链路。

Session 3：万物互联的图数据库设计 (Neo4j)

技术要点：这是最核心的一节课。Mentor 带领大家设计通用的 Person, Event, Media 节点，以及 PARTICIPATED_IN, RELATED_TO 等关系边。

学生任务：在 Neo4j 中建立测试图谱，写出第一条 Cypher 查询语句（例如：“查找所有参与过 1998 年校庆的人”）。

阶段二：AI 注入与知识抽取 (Data Ingestion & Agentic Extraction)
Session 4：多模态感知管道 (Vision & Audio)

技术要点：接入 Whisper API（语音转文字）和视觉大模型 API（如 Gemini Vision 提取照片描述和人脸特征向量）。

学生任务：写一个 Python 脚本，输入一张照片和一段语音，输出结构化的 JSON 描述。

Session 5：信息整理助理 Agent (The Archivist)

技术要点：引入 LangGraph 或 AutoGen 构建无头智能体。将 Session 4 的结果和大模型结合，做实体抽取 (NER) 和同名消歧。

学生任务：实现一个 Pipeline——丢进去一篇会议记录/口述录音，Agent 自动提取三元组并写入 Session 3 的 Neo4j 图数据库。

Session 6：MCP 与工具调用机制

技术要点：讲解如何将 PostgreSQL 的读写和 Neo4j 的查询封装成标准的 MCP Tools。

学生任务：给 Archivist Agent 配备“眼睛和手”，让它能自主判断“遇到新名字时，先去图数据库里查一下有没有同名同姓的”。

阶段三：前端交互与 Vibe Coding 冲刺 (User Experience)
Session 7：统一组件库与 3 套皮肤

技术要点：Mentor 讲解前端组件复用策略。3 个组开始分道扬镳，利用共同的底层 API 开发各自的 UI。

学生任务 (Vibe Coding)：

组 1（学院）：生成带复杂筛选功能的 Web Dashboard。

组 2（校友）：生成时间轴 (Timeline) 风格的 H5 页面。

组 3（家庭）：生成极简的移动端界面，核心是一个巨大的“按住说话/拍照”按钮。

Session 8：对话驱动的交互 (Voice-First/Human-in-the-loop)

技术要点：打通前端语音输入 -> 后端转文本 -> Agent 分析 -> 图谱更新的全链路。

学生任务：实现“智能追问”功能。当老人上传照片说“这是大姑”时，UI 上能弹出 Agent 的温和提问：“大姑旁边穿红衣服的年轻人是谁？”

Session 9：Graph RAG 检索与问答

技术要点：Text-to-Cypher 技术。如何让用户在搜索框输入自然语言，系统去图数据库里找答案。

学生任务：实现各自前端的“智能搜索”功能。

阶段四：艺术生成、本地联通与部署 (Generative Art & Ops)
Session 10：创意艺术家 Agent (The Creator)

技术要点：基于图数据库检索出的“故事线”，组装超长 Prompt，调用视频/音频生成大模型（如 Veo、Suno）。

学生任务：实现“一键生成回忆录”功能。输入一个 Person ID，系统自动调取其所有历年照片，生成带配乐的动态影集或文本故事。

Session 11：OpenClaw 管理者外挂接入（选修/进阶）

技术要点：在管理员的本地电脑部署 OpenClaw。

学生任务：编写 OpenClaw 的本地 Prompt，让它能够自动监控本地硬盘的某个“待上传”文件夹，并将老旧视频自动压缩后通过 API 推送到云端流水线。

Session 12：独立部署、测试与 Demo Day

技术要点：Mentor 讲解 Docker 容器化和云服务器部署（AWS/阿里云）的基础知识。

学生任务：3 个组分别将各自的系统（数据库、后端 Agent、前端）打包上线，进行跨组互相测试（QA），并进行最终的成果路演。
