# AI 原生多模态活档案系统：从架构到落地的 12 周实训大纲

## 前言：三种客户的需求，一种架构
 1）一个学院有 1000 学生，300 教职员工，要不断完善其事件和人物的活档案（照片，视频，论文，文章，学术和非学术活动的各种记录）， 并被学院的各种用户消费（增删改查，并生成各种报告或多模态的艺术展现）。  另外，还有可能被学院内的智能体（如 openClaw 龙虾）调用并输出信息。  这个学院会进一步收集和学校相关的学术界和工业界人物，合作的导师，等 “相关” 人物和发生关系的类型 （合作，推荐学生，演讲，客座教授，...) 这些信息只在本学院可见， 人物的隐私信息只有部分管理人员可见。 
 2）一个历史悠久的校友会， 有很多校友并写了很多回忆文章，并不断有校友聚会。 他们希望把校友的文章、活动、聚会等信息形成丰富的档案并支持增删改查，并生成艺术的展现（文章，周年纪念，短视频等）。 校友可以访问这个档案并不断提供新的回忆或者确认历史和人物的描述。 智能体（如龙虾）也可以逐步爬取网络数据，收集校友信息，核对之后，加入校友会文档。   
 3）一个大家庭， 有老少 4 辈人， 也有很多照片视频和回忆，这些信息少量是文章大量是语言描述， 例如一个老人和孙子一问一答地谈论老照片中的细节。 家庭成员大部分都用手机（不同品牌）， 希望通过手机贡献更多照片，更多语音评论，但是看到成体系的语音+文字对话。 同样， 家庭成员也希望看到艺术方式展现人物的足迹， 某张照片的周年纪念，并分享到外部或者家庭的显示设备上。    
 
 我们不会手动设计 3 种截然不同的架构来实现这三类需求， 而是用一种架构来解决，同时希望达到功能模块的高度重用。 

## 一、 三种客户的需求与产品本质

本项目旨在为三种截然不同的场景构建多模态活档案系统。表面上服务于不同的 B 端（学院、校友会）和 C 端（大家庭）客户，但核心痛点高度一致：**非结构化多模态记忆的沉淀、关联与再创作。**

### 1. 场景剖析

* **场景一：高频生长的“创新引擎”—— 现代学院活档案**
    * **用户画像**：研究员（知识产出者）、学生（高频活跃的流转节点）、教职员工（管理者）。
    * **输入特征**：高频、多模态、强专业性。每日产生大量科研论文（PDF）、组会汇报（PPT）、学术演讲（音视频）及活动合影（图像）。
    * **核心诉求**：构建动态的“科研脉络”与“人才图谱”，实时追踪课题参与情况与技术理念传承。
    * **输出消费**：复杂的增删改查（如调出某教授指导过的学生名单及答辩视频），以及一键生成学术与日常混剪的震撼开场视频。
* **场景二：跨越时空的“情感纽带”—— 历史校友会活档案**
    * **用户画像**：跨越多个届别、遍布全球的校友，存在大量“失联”状态。
    * **输入特征**：低频集中、文本重度、岁月痕迹重。海量长篇回忆录、老照片扫描件及偶尔的大型聚会影像。
    * **核心诉求**：实体消歧（认人）与记忆唤醒。将回忆录中的文字、模糊的老照片人脸与现今的校友精准关联。
    * **输出消费**：校友通过搜索拼凑班级碎片记忆，并在校庆时生成以个人为主角的专属青春交互式纪录片。
* **场景三：随时随地的“口述历史”—— 四世同堂的家族档案**
    * **用户画像**：具有较高技术壁垒的老年人（倾诉者）和习惯移动互联网的年轻一代（记录者）。
    * **输入特征**：移动端优先、语音为主、对话驱动。以“拿着老照片录音问答”为典型交互方式。
    * **核心诉求**：极低阻力的采集与口述历史结构化。系统需直接从方言或口语对话中提取人物关系和历史背景。
    * **输出消费**：将杂乱对话自动整理成“有图有声有文字”的家族时间线并在手机端同步，或在特定节日生成温情的家族传记短片。

### 2. 产品抽象：SaaS/PaaS 平台化切入点

本实训不采用手动设计三套独立架构的模式，而是提取产品最大公约数，构建统一的**“AI 原生多模态图谱引擎 (Multimodal Graph RAG)”**：

* **统一的底层数据结构 (The Universal Ontology)**：底层统一抽象为 `[Person]`、`[Event]` 节点，而论文摘要或口述录音作为挂载的 `[Artifacts]`。
* **统一的“无感”采集引擎 (The Ingestion Layer)**：底层调用同一套多模态处理管道：识别人脸 + 语音转文本 + 提取实体 + 写入图数据库。
* **统一的交互范式 (Human-in-the-loop)**：通过自然的对话进行后台数据纠错与补充（例如主动询问未匹配的人脸信息）。
* **统一的创意引擎 (The Generative Layer)**：Agent 检索精准素材构成故事线，生成 Prompt 调用音视频生成大模型。

交付时，底层架构保持一致，仅通过配置不同的前端交互界面（Web/H5/App）和具有不同权限与设定的数字助理 Agent/创意 Agent，即可完成业务适配。

## 二、 核心架构定调的 5 个关键决策

为确保工程质量，系统设计需严格遵循以下 5 个物理边界与架构规范：

1.  **底层存储的“三足鼎立”**
    * **对象存储 (S3/OSS)**：专门存放原始文件（照片、视频、长文 PDF）。
    * **关系型数据库 (PostgreSQL)**：存放账号体系、权限控制等结构化高价值元数据。
    * **向量数据库 (Vector DB)**：存放高维特征向量，作为实现“同名消歧”和“模糊搜索”的核心引擎。
2.  **关系数据的图谱化 (Graph RAG)**
    * 必须使用图数据库（如 Neo4j）存储实体节点和关系边。严禁使用 Markdown 等无结构文本存储复杂关联网络，以避免大模型上下文超载和严重幻觉。
3.  **Voice-First Hybrid UI 前端交互**
    * 前端形态必须是“对话与传统 UI 的深度融合”。针对长者提供“大按钮语音+照片瀑布流”的极简采集端；针对重度用户提供传统组件（时间轴、关系图）配合悬浮智能对话框的消费端。
4.  **确定性与非确定性模块的通信隔离**
    * **传统 Web Service (RESTful/gRPC)**：处理系统内部确定的数据流转（如文件上传、鉴权）。
    * **MCP (Model Context Protocol) / Skill**：专用于 AI 智能体任务流，为大模型提供受控的数据库查询和工具调用权限。
5.  **OpenClaw 的精准定位**
    * **禁止项**：不可作为系统高并发后端，不可面向终端用户开放。
    * **适用项**：仅部署于系统管理员的本地工作站，作为“本地素材归集中转站”或调用本地重型剪辑软件（如 Premiere）的桌面级自动化外挂。

## 三、 12 次 MVP 冲刺实训大纲 (The 12-Session Sprints)

由资深 DB 工程师和前端工程师作为 Mentor 带领，3 组具有初级 Vibe Coding 经验的学生团队将在 12 次实训中各自开发出 1 个 MVP，共享底层设计与模块，独立完成部署。

### 阶段一：底层基建与数据建模 (Infrastructure & Ontology)
* **Session 1：产品宣发与架构破冰**
    * **技术要点**：拆解 3 个场景的共性，发布架构白皮书与规范。
    * **学生任务**：配置开发环境，跑通 Cursor/Windsurf 基础工作流。
* **Session 2：数据库搭建与 API 契约**
    * **技术要点**：演示 PostgreSQL + 对象存储 + 向量数据库的集成。
    * **学生任务**：利用 Vibe Coding 生成后端基础 CRUD RESTful API，打通本地至对象存储的上传链路。
* **Session 3：图数据库设计核心 (Neo4j)**
    * **技术要点**：设计通用的 `Person`、`Event`、`Media` 节点，及 `PARTICIPATED_IN` 等关系边。
    * **学生任务**：在 Neo4j 建立测试图谱，编写核心 Cypher 查询语句。

### 阶段二：AI 注入与知识抽取 (Data Ingestion & Agentic Extraction)
* **Session 4：多模态感知管道 (Vision & Audio)**
    * **技术要点**：接入 Whisper API 与视觉大模型 API。
    * **学生任务**：编写 Python 脚本，实现将单张照片和语音转化为结构化 JSON 描述。
* **Session 5：信息整理助理 Agent (The Archivist)**
    * **技术要点**：引入 LangGraph/AutoGen 构建无头智能体，结合前序成果进行实体抽取 (NER) 和同名消歧。
    * **学生任务**：实现非结构化数据流入 -> Agent 提取三元组 -> 写入 Neo4j 图数据库的自动化 Pipeline。
* **Session 6：MCP 与工具调用机制**
    * **技术要点**：将 PostgreSQL 读写和 Neo4j 查询封装为标准 MCP Tools。
    * **学生任务**：为 Archivist Agent 配备数据库查询“技能”，使其具备在抽取数据前先进行库内检索比对的自主判断能力。

### 阶段三：前端交互与 Vibe Coding 冲刺 (User Experience)
* **Session 7：统一组件库与 3 套 UI 皮肤**
    * **技术要点**：讲解前端组件复用策略，基于共同底层 API 开启分支开发。
    * **学生任务 (Vibe Coding)**：
        * 组 1（学院）：生成带复杂筛选功能的 Web Dashboard。
        * 组 2（校友）：生成时间轴 (Timeline) 风格的交互页面。
        * 组 3（家庭）：生成极简移动端界面（巨大语音采集按钮优先）。
* **Session 8：对话驱动的交互 (Human-in-the-loop)**
    * **技术要点**：打通前端语音 -> 转文本 -> Agent 分析 -> 图谱更新的全链路。
    * **学生任务**：实现“智能追问”机制。当上传信息缺失时，UI 自动弹出 Agent 的上下文关联提问。
* **Session 9：Graph RAG 检索与可视化问答**
    * **技术要点**：应用 Text-to-Cypher 技术。
    * **学生任务**：在各自前端实现基于自然语言的智能搜索与图谱结果展现。

### 阶段四：艺术生成、本地联通与部署 (Generative Art & Ops)
* **Session 10：创意艺术家 Agent (The Creator)**
    * **技术要点**：基于图谱检索“故事线”组装超长 Prompt，调用音视频生成 API (如 Veo)。
    * **学生任务**：实现“一键生成回忆录”功能（输入 Person ID 生成配乐动态影集）。
* **Session 11：OpenClaw 管理者外挂接入 (高级实战)**
    * **技术要点**：在特定受控的本地环境中部署 OpenClaw。
    * **学生任务**：编写本地控制指令，使 OpenClaw 自动监控特定文件夹，执行预处理并批量推送至云端流水线。
* **Session 12：独立部署、测试与 Demo Day**
    * **技术要点**：Docker 容器化与云服务器 (AWS/阿里云) 部署精讲。
    * **学生任务**：各组完成系统打包上线，执行交叉 QA，完成产品路演。

## 四、 教学成果与技术价值观总结

通过这 12 周的高强度工程演练，学生不仅各自交付了具备商业雏形的 MVP，更建立了一套成熟的**AI 软件工程（AI-Native Software Engineering）方法论**。他们深刻理解了各项新兴技术的优缺点，避免了为了“炫技”而滥用技术的陷阱：

1.  **重新定义 Vibe Coding 的边界**：学生充分体验了利用 AI 辅助编程（如 Cursor）一键生成前端 UI 和样板代码的极速快感。但由于 Mentor 在前期（Session 2 & 3）严格锁死了 API 契约和数据库表结构，学生深刻意识到：**AI 可以代写代码，但不能代做架构设计**。扎实的软件工程规范是防止 Vibe Coding 演变成不可维护的“面条代码”的唯一屏障。
2.  **认清“确定性”与“非确定性”的界限**：学生学会了不滥用对话框。他们意识到，对于收集表单等确定的结构化输入，传统 Web UI 依然是效率之王；而纯文本聊天机器人并不适合展示复杂的时间轴和人物脉络。因此，系统最终走向了“传统 UI 负责展示与精准控制，AI 对话框负责模糊意图处理与信息补全”的混合体验架构。
3.  **MCP 与 Skill 的精准应用**：通过引入 MCP（Session 6），学生理解了如何让大模型与真实的工程世界安全对话。不再是盲目地给大模型开放全局后门，而是通过标准协议暴露受限的查询“工具（Skills）”，将 AI 真正规训为业务流程中的一环。
4.  **扎实的 Cloud-Native 数据底座**：跨越了玩具级单机应用的局限。学生熟练掌握了将对象存储（S3）、关系型数据（PostgreSQL）与非关系型数据（Vector/Neo4j）解耦融合的“三足鼎立”架构，以及最终的 Docker 容器化部署，具备了企业级 Cloud 基础。
5.  **克制而精准地使用 OpenClaw**：打破了“万物皆可本地 Agent”的迷思。学生明确了 OpenClaw 的生态定位——它不是用来替代高并发的云端后端服务的，也不是面向海量普通用户的。它唯一的、无可替代的价值在于打通**物理环境与云端**的最后一公里（如调用 OS 级别的本地文件系统和重型软件）。

总而言之，本课程不仅教会了学生如何组合最新的 AI 工具，更教会了他们作为未来工程师最核心的素质：**在深刻理解技术本质的基础上，进行克制、精准且富有创造力的架构选型。**
